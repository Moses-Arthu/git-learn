LESSON 01 · FOUNDATIONS
What is Git?
Git is a version control system — a tool that tracks every change you make to your code, so you can go back in time, work in parallel, and collaborate with others without fear.
Analogy: Think of Git like Google Docs' version history, but for your entire project folder. Every time you hit "save" (called a commit), Git takes a snapshot. You can always go back to any snapshot.
Why do developers use Git?

🕰 Time travel

Made a mistake? Broke something that was working? Git lets you instantly go back to any previous version of your code.

🌿 Parallel work

Work on a new feature without touching the stable version of your code, using branches. Merge when it's ready.

🤝 Collaboration

Multiple developers can work on the same project simultaneously. Git merges everyone's work together and flags conflicts.

Git vs GitHub — not the same thing!
Git = the tool installed on your computer. It tracks changes locally. GitHub = a website that hosts your Git repos online so you can share & backup. GitLab / Bitbucket = other websites that do the same thing as GitHub.

LESSON 02 · FOUNDATIONS
Setup & Configuration
Before using Git, you install it and tell it who you are. Git will attach your name and email to every commit you make.
1. Install Git
terminal
# Check if Git is already installed
git --version
git version 2.43.0

# If not installed:
# Windows: download from https://git-scm.com
# macOS: brew install git
# Linux: sudo apt install git
2. Tell Git who you are (one-time setup)
terminal
git config --global user.name "Moses"
git config --global user.email "moses@example.com"

# Verify your settings
git config --list
user.name=Moses
user.email=moses@example.com

💡 What does --global mean?

It applies this setting to all repositories on your computer. You only need to do this once. Without --global, the setting only applies to the current repo.

LESSON 03 · FOUNDATIONS
Your First Repository
A repository (repo) is a folder that Git is tracking. You either create a new one with git init or copy an existing one with git clone.
Analogy: Running git init is like placing a surveillance camera in your project folder. From that moment on, Git watches everything that changes inside it.
Creating a new repo from scratch
terminal
# Create a new folder for your project
mkdir my-project
cd my-project

# Turn it into a Git repository
git init
Initialized empty Git repository in /my-project/.git/

# Check the status (nothing tracked yet)
git status
On branch main
No commits yet
Cloning an existing repo
terminal
# Download a repo from GitHub to your machine
git clone https://github.com/username/project.git
Cloning into 'project'...
done.

📁 What is the .git folder?

When you run git init, Git creates a hidden .git/ folder inside your project. This is where Git stores all the history and tracking data. Never delete it manually.

LESSON 04 · FOUNDATIONS
Staging & Commits
This is the core of Git's daily workflow. You make changes, stage what you want to save, then commit to lock in a snapshot.
Analogy: Think of it like packing a box to ship. Your changed files sit on the floor (working directory). You pick some up and put them in the box (staging area). When you tape the box shut, that's a commit — a permanent snapshot.
Working Directory ──git add──▶ Staging Area ──git commit──▶ Repository (your files) (the box) (history)
The full workflow
terminal
# 1. Create or edit a file
echo "Hello Git" > index.html

# 2. Check what changed
git status
Untracked files: index.html

# 3. Stage the file (add to the box)
git add index.html
# Or stage ALL changed files:
git add .

# 4. Check status again
git status
Changes to be committed: index.html

# 5. Commit (tape the box shut with a message)
git commit -m "Add homepage"
[main (root-commit) a1b2c3d] Add homepage
1 file changed, 1 insertion(+)

# 6. View commit history
git log --oneline
a1b2c3d Add homepage

✍️ Writing good commit messages

  • Use present tense: "Add login page" not "Added login page"
  • Be specific: "Fix null check in auth module" not "Fix bug"
  • Keep it under 72 characters

LESSON 05 · BRANCHING
Branches
Branches let you work on new features or fixes in isolation, without breaking the main working code. They're one of Git's most powerful features.
Analogy: Imagine your codebase is a tree trunk (main). When you want to try something new, you grow a branch off the side. The trunk is unaffected until you decide to merge the branch back.
main: A ── B ── C ────────────────── F
\ feature: D ── E ──────────/
Branch commands
terminal
# See all branches (* = current branch)
git branch
* main

# Create a new branch
git branch feature/login

# Switch to it
git checkout feature/login
# Modern shortcut (create + switch at once):
git checkout -b feature/login

# Work on it, add commits...
git add .
git commit -m "Add login form"

# Switch back to main when done
git checkout main

🏷 Branch naming conventions

  • feature/user-auth — new feature
  • fix/login-crash — bug fix
  • hotfix/payment-error — urgent fix
  • main or master — stable production code

LESSON 06 · BRANCHING
Merging & Conflicts
When your feature branch is ready, you merge it back into main. Sometimes Git can do this automatically; sometimes it needs your help to resolve a conflict.
Merging a branch
terminal
# Step 1: Switch to the branch you want to merge INTO
git checkout main

# Step 2: Merge the feature branch into main
git merge feature/login
Updating a1b2c3d..f4e5d6c
Fast-forward
login.html | 20 ++++

# Step 3: Delete the branch (optional, it's done)
git branch -d feature/login
Handling merge conflicts

⚡ What is a conflict?

A conflict happens when two branches changed the same line of the same file differently. Git doesn't know which version to keep — it asks you to decide.

terminal — conflict markers in your file
<<<<<<< HEAD (main)
Hello World
=======
Hello Git
>>>>>>> feature/login

# Fix: edit the file to keep what you want, then:
git add index.html
git commit -m "Resolve merge conflict"

LESSON 07 · REMOTE
GitHub & Remotes
A remote is a version of your repo hosted online (like GitHub). You push to send your local commits up, and pull to receive new commits from others.
Your machine ────push────▶ GitHub (remote) Your machine ◀────pull──── GitHub (remote)
Connecting to GitHub
terminal
# After creating a repo on GitHub, link it:
git remote add origin https://github.com/you/project.git

# See your remotes
git remote -v
origin https://github.com/you/project.git (fetch)
origin https://github.com/you/project.git (push)

# Push your commits to GitHub for the first time
git push -u origin main
Branch 'main' set up to track remote branch 'main' from 'origin'.

# Future pushes (after the first) are just:
git push

# Pull latest changes from GitHub
git pull

🔑 What is origin?

origin is just the conventional nickname for your primary remote (usually GitHub). You could name it anything, but everyone uses origin by convention.

📤 The typical team workflow

  • git pull — get latest changes from the team
  • Make your edits on a feature branch
  • git add . && git commit -m "..."
  • git push — share your work
  • Open a Pull Request on GitHub for review
